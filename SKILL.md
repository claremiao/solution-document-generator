# 信息化解决方案文档生成技能

## 技能概述

本技能是面向 IT 信息化系统集成建设项目的专业文档编制工具，能够根据用户需求自动生成符合行业规范的信息化解决方案文档。技能覆盖从需求采集、大纲生成、目录定制到正文编写的完整流程，确保输出内容专业、严谨、逻辑性强。

**适用场景：**
- 编制信息化解决方案
- 撰写项目建议书
- 编写可行性研究报告
- 准备投标技术方案
- 生成系统建设方案
- 编制数字化转型规划

**使用时机：**
- 用户提供了项目基本信息，需要生成完整的解决方案文档
- 需要快速编制符合专业规范的信息化项目文档
- 需要确保文档内容完整、结构清晰、专业严谨
- 需要生成可指导项目实施的专业方案

## 核心工作流程

本技能采用四阶段工作流程，确保文档编制的专业性和完整性：

```
需求采集 → 大纲生成 → 目录定制 → 正文编写 → 质量检查
```

### 流程说明

1. **需求信息采集**：通过结构化模板引导用户提供项目核心信息
2. **大纲生成**：基于需求自动生成符合规范的多层级目录大纲
3. **目录定制**：允许用户灵活调整目录结构直至满意
4. **正文编写**：按确认的目录结构逐章编写专业内容
5. **质量检查**：验证内容完整性、准确性和可操作性

## 第一步：需求信息采集

### 采集方式

提供三种需求采集方式，用户可根据情况选择：

**方式一：交互式问答（推荐）**

执行脚本进行引导式问答：

```bash
node scripts/collect_requirements.cjs
```

脚本将按以下类别逐一提问：
- 项目背景（行业现状、政策依据、痛点挑战）
- 目标用户（用户群体、规模、使用场景）
- 核心业务需求（业务流程、关键功能、数据需求）
- 技术要求（架构偏好、技术栈、性能指标）
- 预算范围（总投资、软件、硬件、运维）
- 实施周期（启动时间、关键节点、总周期）
- 特殊约束（合规要求、技术限制、资源约束）

**方式二：直接提供信息**

用户按以下结构直接提供项目信息：

```
项目名称：[填写]
所属行业：[填写]
项目背景：[描述行业现状、政策依据、面临痛点]
目标用户：[描述用户群体、规模、使用场景]
核心需求：[描述主要业务流程、关键功能]
技术要求：[描述架构、技术栈、性能要求]
预算范围：[填写预算金额]
实施周期：[填写时间要求]
特殊约束：[填写合规、技术等约束条件]
```

**方式三：混合模式**

用户先提供基础信息，AI 助手通过追问补充细节。

### 关键采集要点

**项目背景**
- 行业发展现状与趋势
- 相关政策依据（如《中国制造 2025》《"十五五"规划》等）
- 企业当前信息化水平
- 面临的主要痛点和挑战

**目标用户**
- 用户角色分类（如操作工、班组长、管理层）
- 用户规模预估（并发用户数、总用户数）
- 典型使用场景（现场操作、办公室分析、移动审批）

**业务需求**
- 核心业务流程描述
- 关键业务功能清单
- 数据处理需求（采集、存储、分析）
- 系统集成需求（与 ERP、MES 等系统对接）

**技术要求**
- 架构偏好（微服务/单体、云原生/传统）
- 技术栈要求（开发语言、数据库、中间件）
- 性能指标（并发量、响应时间、可用性）
- 安全要求（等保级别、数据加密、权限控制）

**预算与周期**
- 总投资预算及分项预算
- 项目启动时间和关键节点
- 项目总工期要求
- 是否分期建设

**约束条件**
- 合规性要求（等保、行业规范）
- 技术限制（需兼容现有系统）
- 资源约束（利旧、人员限制）
- 其他特殊要求

### 信息验证

采集完成后，应进行信息完整性检查：

✓ 项目背景清晰，有明确的业务驱动因素
✓ 目标用户明确，用户规模可量化
✓ 核心业务流程清晰，关键功能可识别
✓ 技术要求具体，性能指标可衡量
✓ 预算范围合理，与功能规模匹配
✓ 实施周期可行，关键节点明确
✓ 约束条件完整，无重大遗漏

如信息不完整，应通过追问补充。

### 大纲定制规则

根据项目特点自动调整大纲：

**微服务架构项目**
- 在第 3 章增加"3.6 微服务治理方案"
- 在第 4 章增加容器化、服务网格相关内容

**高安全要求项目**
- 在第 6 章增加"6.6 安全等级保护方案"
- 在第 3 章强化安全架构设计内容

**大型复杂项目**
- 可增加"数据治理设计"独立章节
- 可增加"集成设计"独立章节
- 可增加"风险评估"独立章节

**小型精简项目**
- 可精简为 5 章：项目概述、需求分析、总体设计、详细设计、实施计划
- 合并质量保障、运维支持到实施计划章节

### 生成命令

使用脚本生成大纲：

```bash
node scripts/generate_outline.cjs
```

输出格式：
- Markdown 格式：便于阅读和编辑
- JSON 格式：便于程序处理和定制

## 第三步：目录自定义

### 支持的自定义操作

提供 6 种目录调整操作：

**1. 添加章节**

在指定位置添加新章节：

```
操作：添加章节
父章节：第 4 章 详细设计
新章节名称：移动端应用设计
子章节：
  - 移动端架构设计
  - 用户界面设计
  - 数据同步设计
  - 安全认证设计
位置：第 4.3 节之后
```

**2. 删除章节**

删除不需要的章节：

```
操作：删除章节
章节编号：6.5
确认：是
理由：持续改进机制内容可并入质量保证措施
```

**3. 修改章节名称**

调整章节标题使其更准确：

```
操作：修改章节名称
章节编号：4.1
原名称：功能模块一设计
新名称：生产管理模块设计
```

**4. 调整章节顺序**

改变章节排列顺序：

```
操作：调整顺序
章节编号：第 8 章
目标位置：移至第 7 章之前
理由：投资估算应在运维支持之前
```

**5. 合并章节**

合并多个章节为一个：

```
操作：合并章节
源章节：4.1, 4.2, 4.3
合并后名称：业务功能模块设计
子章节处理：重新组织为 5 个子章节
```

**6. 拆分章节**

将大章节拆分为多个章节：

```
操作：拆分章节
源章节：第 4 章 详细设计
拆分方式：按业务域分组
新章节 1：业务功能设计（4.1-4.3）
新章节 2：技术实现设计（4.4-4.5）
```

### 验证规则

每次调整后自动验证：

✓ **必需章节检查**：项目概述、需求分析、总体设计、详细设计、实施计划必须存在
✓ **子章节数量**：每个主章节 3-8 个子章节
✓ **编号连续性**：章节编号连续无跳号
✓ **逻辑合理性**：章节顺序符合认知逻辑

### 模板选择

提供多种模板供快速选择：

**标准模板**：8 章结构，适用于大多数项目
**精简模板**：5 章结构，适用于小型项目
**大型项目模板**：10+ 章结构，适用于复杂项目
**行业模板**：政务、制造、医疗等行业特定模板

## 第四步：正文内容编写

### 编写原则

**内容深度要求**
- 根据项目复杂度及金额，调整内容深度，500万以上项目每个大章节可扩展到4-5层子章节
- 每个子章节在300-400 字之间
- 核心章节 800-1500 字
- 信息充实，避免空洞描述
- 关键要点详细阐述，避免模糊描述

**内容结构**
- 开篇概述（50-100 字）：说明本节内容和目的
- 主体内容（300-1200 字）：分 3-5 个要点详细阐述
- 总结说明（50-100 字）：小结或过渡

**写作风格**
- 专业、正式、
- 逻辑清晰：层次分明，论证充分
- 数据支撑：关键论点有数据或事实支撑
- 可操作性：提供具体实施指导

### 各章节编写要点

#### 第 1 章 项目概述

**1.1 项目背景**
- 行业发展现状与趋势（300-500 字）
- 政策环境分析（300-500 字）
- 企业现状与挑战（现状分析300字、挑战分析300-500字）

示例内容应包含：
- 行业数据引用（如"数字化转型渗透率达到 45%"）
- 政策文件引用（如"《中国制造 2025》"）
- 具体痛点描述（如"数据孤岛严重、质量追溯困难"）

**1.2 建设目标**
- 总体目标（200-300 字）
- 具体目标（200-300 字）：业务目标、技术目标、管理目标
- 目标实现路径（300-500 字）

目标应遵循 SMART 原则，量化指标明确基准值和目标值。

**1.3 建设范围**
- 业务范围（200-300 字）
- 系统范围（200-300 字）
- 组织范围（200-300 字）

**1.4 建设原则**
- 技术原则（200-300 字）
- 业务原则（200-300 字）
- 实施原则（200-500 字）

**1.5 术语定义**
- 以表格形式列出专业术语
- 提供准确定义和说明

#### 第 2 章 需求分析

**2.1 业务需求分析**
- 业务现状描述（200-300 字）
- 业务改进需求（200-250 字）
- 业务协同需求（200-300 字）

使用业务流程图辅助说明，量化业务指标。

**2.2 功能需求分析**
- 功能需求概述（200-300 字）
- 核心功能需求（250-350 字）
- 辅助功能需求（200-300 字）

每个功能按以下模板描述：
```
功能名称：[填写]
功能描述：[填写]
用户需求：[填写]
输入：[填写]
处理：[填写]
输出：[填写]
验收标准：[填写]
```

**2.3 非功能需求分析**
- 性能需求（200-300 字）：并发量、响应时间、处理能力
- 安全需求（200-300 字）：认证、授权、加密、审计
- 可靠性需求（200-300 字）：可用性、容灾、恢复
- 可维护性需求（200-300 字）：监控、日志、配置

**2.4 用户需求分析**
- 用户角色分类（200-300 字）
- 用户使用场景（200-250 字）
- 用户体验需求（200-300 字）

**2.5 需求优先级矩阵**
- 优先级划分标准（200-300 字）
- 需求优先级列表（200-250 字）
- 分期实施建议（200-300 字）

#### 第 3 章 总体设计

**3.1 总体架构设计**
- 架构设计目标（300-400 字）
- 系统架构（300-500 字）
> 系统架构包括系统组件、系统模块、系统服务等。配置系统架构图。
- 业务架构（300-500 字）
> 业务架构包括业务流程、业务模块、业务服务等。配置业务架构图。
- 集成架构（300-500 字）
> 集成架构包括系统组件之间的集成、系统模块之间的集成等。配置集成架构图。
- 数据架构（300-500 字）
> 数据架构包括数据存储、数据处理、数据交互等。配置数据架构图。

配合架构图进行说明，清晰描述各层次关系。

**3.2 技术架构设计**
- 技术架构概述（300-500 字）
- 架构风格设计（300-500 字）：展现层、应用层、数据层、基础设施层
> 架构风格包括系统组件的分布、系统模块的组织、系统服务的部署等。配置架构图。
- 关键技术设计（300-500 字）：微服务、接口、中间件等。

**3.3 技术栈选型方案**
- 选型原则（300-500 字）
- 核心技术栈（300-500 字）
- 技术对比分析（300-500 字）
> 技术对比分析包括技术栈的性能、可扩展性、可维护性等。配置技术对比分析图。

使用表格展示技术选型：
```
| 技术类别 | 候选方案 | 最终选择 | 选择理由 |
|----------|----------|----------|----------|
| 后端框架 | Spring Boot / Node.js | Spring Boot | 生态成熟、团队熟悉 |
| 数据库 | MySQL / PostgreSQL | MySQL | 性能优异、文档丰富 |
```

**3.4 系统边界与接口**
- 系统边界定义（300-500 字）
- 内部接口设计（300-500 字）
- 外部接口设计（300-500 字）

**3.5 部署架构设计**
- 部署环境要求（300-500 字）
> 部署环境要求包括服务器配置、数据库配置、网络配置等。配置部署环境图。
- 部署方案设计（300-500 字）
> 部署方案设计包括系统组件的部署、系统模块的部署、系统服务的部署等。配置部署方案图。
- 高可用设计（300-500 字）
> 高可用设计包括系统组件的高可用、系统模块的高可用等。配置高可用图。

#### 第 4 章 详细方案设计

每个功能模块按以下结构编写：

**4.1 功能模块设计**
- 方案概述（300-500 字）：定位、功能、价值
- 方案亮点（300-500 字）：技术优势、业务优势、成本优势等
> 紧密结合客户信息化建设现状、业务痛点，突出解决方案的优势。有逻辑，有说服力
- 功能设计（300-500 字）：功能列表、详细设计、流程图
> 功能设计包括系统组件的功能、系统模块的功能、系统服务的功能等。配置功能图。
- 数据设计（300-500 字）：数据结构、流转、存储
> 数据设计包括数据存储、数据处理、数据交互等。配置数据流图。

**4.2 数据库设计**

- 数据库设计原则（300-500 字）
- 需求分析（300-500 字）
> 数据库设计需求分析包括数据存储、数据处理、数据交互等。配置需求分析图。
- 概念模型设计（300-500 字）：E-R 图说明
- 逻辑模型设计（300-500 字）：核心表结构
> 结合业务需求场景设计2-3个库表结构，结合业务，有逻辑，可落地实现。
- 物理设计（300-500 字）：存储引擎、分区策略
> 数据库物理设计包括存储引擎、分区策略等。配置物理设计图。

## 常见问题

### Q1: 如何处理用户提供的信息不完整？

**A:** 采用追问策略补充信息：
1. 识别缺失的关键信息
2. 针对性地提出问题
3. 提供选项或示例引导
4. 记录假设条件（如用户无法提供）

### Q2: 技术方案应该多详细？

**A:** 详细到可指导实施，但避免过度细节：
- 说明技术选型理由
- 描述架构设计思路
- 提供关键技术实现方案
- 不包含具体代码实现

### Q3: 如何确保内容的专业性？

**A:** 遵循以下原则：
- 使用行业标准术语
- 引用权威标准规范
- 参考行业最佳实践
- 关键方案有理论或实践依据

### Q4: 如何处理不确定的信息？

**A:** 采取以下策略：
- 明确标注"待定"或"需进一步确认"
- 提供初步建议方案
- 说明假设条件
- 避免给出不准确承诺

### Q5: 文档篇幅应该多长？

**A:** 根据项目规模确定：
- 小型项目：30-50 页（1.5-2.5 万字）
- 中型项目：50-80 页（2.5-4 万字）
- 大型项目：80-150 页（4-7.5 万字）
- 核心是内容充实，避免凑字数
