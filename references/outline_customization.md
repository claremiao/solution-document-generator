# 目录大纲自定义功能指南

## 功能概述

本功能提供灵活的目录结构调整能力，允许用户根据具体项目需求对自动生成的标准目录进行个性化定制。

## 支持的自定义操作

### 1. 添加章节

**使用场景：**
- 项目有特殊功能模块需要单独成章
- 行业规范要求增加特定章节
- 用户有额外的关注点需要强调

**操作方式：**
```
操作指令：添加章节
父章节：选择要添加到的主章节（如"详细设计"）
新章节名称：输入新章节标题
子章节列表：输入该章节包含的子章节（3-5 个）
位置：指定插入位置（末尾或指定章节之前）
```

**示例：**
```
在"详细设计"章节下添加"移动端设计"子章节
包含子章节：
- 移动端架构设计
- 用户界面设计
- 离线功能设计
- 数据同步设计
- 安全认证设计
```

### 2. 删除章节

**使用场景：**
- 某些章节内容不适用于当前项目
- 项目规模较小，部分章节可合并
- 用户明确要求精简文档结构

**操作方式：**
```
操作指令：删除章节
章节编号：输入要删除的章节编号（如 4.3）
确认：确认删除操作（防止误删）
```

**注意事项：**
- 删除主章节时，其下所有子章节将一并删除
- 每个主章节至少保留 3 个子章节
- 核心章节（项目概述、需求分析、总体设计、详细方案设计、实施计划）不可删除

### 3. 修改章节名称

**使用场景：**
- 章节名称不够准确或专业
- 需要更符合行业术语
- 与用户内部命名规范保持一致

**操作方式：**
```
操作指令：修改章节名称
章节编号：输入要修改的章节编号
原名称：显示当前名称
新名称：输入新的章节标题
```

**示例：**
```
将"功能模块一设计"修改为"生产管理模块设计"
将"功能模块二设计"修改为"质量管理模块设计"
```

### 4. 调整章节顺序

**使用场景：**
- 调整内容逻辑顺序
- 突出重要内容
- 符合特定阅读习惯

**操作方式：**
```
操作指令：调整章节顺序
章节编号：输入要移动的章节编号
目标位置：指定移动到的位置（如"移至第 3 章之后"）
```

**示例：**
```
将"投资估算"章节移至"实施计划"之前
将"运维与支持"章节移至最后
```

### 5. 合并章节

**使用场景：**
- 小型项目内容较少
- 相关章节内容紧密关联
- 精简文档篇幅

**操作方式：**
```
操作指令：合并章节
源章节列表：输入要合并的章节编号
合并后名称：输入新章节名称
子章节处理：选择保留、合并或重新定义
```

**示例：**
```
合并"软件投资估算"和"硬件投资估算"为"系统投资估算"
```

### 6. 拆分章节

**使用场景：**
- 章节内容过多需要拆分
- 部分内容重要性较高需独立成章
- 提高内容组织的清晰度

**操作方式：**
```
操作指令：拆分章节
源章节编号：输入要拆分的章节编号
拆分方式：按子章节分组拆分
新章节名称：为拆分后的章节命名
```

**示例：**
```
将"详细设计"拆分为：
- "业务功能设计"（包含业务模块相关子章节）
- "技术架构设计"（包含技术实现相关子章节）
```

## 自定义操作流程

### 标准流程

1. **查看当前目录**：显示完整的目录结构
2. **选择操作类型**：从 6 种操作中选择
3. **执行操作**：按提示完成具体操作
4. **预览变更**：显示调整后的目录结构
5. **确认或撤销**：确认变更或撤销操作
6. **继续调整**：选择是否继续其他调整
7. **完成确认**：确认最终目录结构

## 目录结构验证规则

### 结构完整性检查

1. **必需章节检查**
   - 必须包含：项目概述、需求分析、总体设计、详细方案设计、实施计划
   - 建议包含：质量保障、运维支持、投资估算

2. **子章节数量与深度检查**
   - 每个主章节：3-8 个子章节
   - 章节深度：3-5 层（如 1.1 → 1.1.1 → 1.1.1.1）
   - 过少（<3）：提示合并或补充
   - 过多（>8）：提示拆分

3. **编号连续性检查**
   - 章节编号连续无跳号
   - 子章节编号格式正确（如 1.1, 1.2, 1.3）
   - 多层级编号格式正确（如 1.1.1, 1.1.2, 1.1.1.1）

4. **逻辑合理性检查**
   - 章节顺序符合认知逻辑
   - 相关内容组织在一起
   - 无明显的内容交叉重复

### 验证输出示例

```
=== 目录结构验证报告 ===

✓ 结构完整性：通过
✓ 子章节数量：通过
✓ 编号连续性：通过
✓ 逻辑合理性：通过

验证结果：目录结构符合规范
```

## 最佳实践建议

### 1. 目录结构设计原则

- **逻辑清晰**：章节间逻辑关系明确，层次分明
- **重点突出**：核心内容独立成章，避免淹没在细节中
- **适度细化**：子章节粒度适中，避免过粗或过细
- **便于阅读**：符合读者认知习惯，便于快速定位

### 2. 常见调整场景

**场景一：功能模块较多**
```
建议：在"详细设计"下按业务域分组
- 生产管理模块组（包含 3-5 个子模块）
- 质量管理模块组（包含 3-5 个子模块）
- 设备管理模块组（包含 3-5 个子模块）
```

**场景二：技术复杂度高**
```
建议：增加"技术专项设计"章节
- 微服务架构设计
- 分布式数据库设计
- 高并发处理设计
- 容灾备份设计
```

**场景三：安全要求严格**
```
建议：增加"安全设计"独立章节
- 安全体系架构
- 身份认证设计
- 访问控制设计
- 数据加密设计
- 安全审计设计
```

### 3. 命名规范

**章节命名：**
- 使用名词性短语，如"总体架构设计"
- 避免动词开头，如"设计总体架构"
- 保持同级章节命名风格一致

**子章节命名：**
- 具体明确，如"用户管理模块设计"
- 避免过于抽象，如"模块设计"
- 体现内容特征，便于快速理解

## 版本管理

### 目录版本记录

每次自定义调整后应记录：
- 版本号（v1.0, v1.1, v2.0）
- 调整内容描述
- 调整时间
- 调整原因

### 版本对比

支持对比不同版本的目录结构：
- 新增章节
- 删除章节
- 修改章节
- 顺序变化

## 常见问题

### Q1: 如何确定合适的章节数量？
**A:** 一般 5-10 个主章节为宜。少于 5 章可能内容组织不够清晰，多于 10 章可能过于复杂。根据项目规模和复杂度灵活调整。

### Q2: 子章节是否可以再分子章节？
**A:** 本技能支持多级结构，章节深度可达 3-5 层（如 1.1 → 1.1.1 → 1.1.1.1）。根据项目复杂度及金额调整深度，500万以上项目每个大章节可扩展到4-5层子章节。小型项目建议保持2-3层结构。

### Q3: 能否自定义章节编号规则？
**A:** 默认使用"章 - 节"编号（如 1.1）。如需其他编号规则（如一、1.1、1.1.1），可在导出时选择相应格式。

### Q4: 调整后能否恢复到标准模板？
**A:** 可以。提供"重置为标准模板"功能，但会丢失所有自定义调整，需谨慎使用。

### Q5: 如何保存多个目录版本？
**A:** 支持保存多个版本方案。用户可创建"方案 A"、"方案 B"等进行对比选择。
